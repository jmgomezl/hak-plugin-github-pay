# Repository Guidelines

## Project Structure & Module Organization
- `src/` holds the TypeScript source. `src/index.ts` is the public library barrel (default export = the HAK plugin); `src/plugin.ts` defines the plugin object and `githubPayPluginToolNames`.
- `src/config.ts` resolves runtime config from the HAK `Context` (`resolveGithubPayConfig`); `src/networks.ts` holds network endpoints and topic metadata; `src/types.ts` defines shared types.
- `src/tools/` implements one agent tool per file (each extends `GithubPayTool` in `src/tools/base.ts`).
- `src/hcs.ts` (topic provisioning, `store.json`, mirror reads), `src/resolve.ts` (fold topics into the current view), `src/pay.ts` (transfer + receipt + provenance), and `src/csv.ts` are the shared domain logic.
- `src/server.ts` is the Express webhook server (`createWebhookServer`, HMAC, `/health`); `src/agent.ts` is the Gemini 2.5 Flash function-calling loop; `src/main.ts` is the runnable app (also the `github-pay` bin).
- `tests/` contains Vitest suites (`*.test.ts`). `dist/` is generated by `tsup`; do not edit by hand.
- Root configs: `tsconfig.json`, `tsup.config.ts`, `biome.json`, `package.json`.

## Build, Test, and Development Commands
- `npm run dev` runs `tsup --watch` for local iteration.
- `npm run build` emits dual ESM/CJS bundles + types to `dist/`.
- `npm run typecheck` runs `tsc --noEmit` over `src/` and `tests/`.
- `npm run lint` runs Biome checks; `npm run format` applies formatting.
- `npm run test` runs Vitest once; `npm run test:watch` keeps it in watch mode.
- `npm start` runs the built app (`node dist/main.js`) — provisions topics and starts the webhook server.

## Coding Style & Naming Conventions
- TypeScript with ESM (`"type": "module"`). Biome enforces 2-space indentation and 100-char lines.
- `camelCase` for variables/functions, `PascalCase` for types/classes, `UPPER_SNAKE_CASE` for constants.
- File names are lower-camelCase in `src/tools/`; scripts use `kebab-case`.
- Tool methods are namespaced `github_pay_*` (collision-safe and valid as Gemini function names); the human-readable `name` is `"GitHub Pay: …"`.
- New tools: mirror an existing file in `src/tools/`, read config via `resolveGithubPayConfig(context)`, and export both the class and a singleton; then register it in `src/plugin.ts` and `src/index.ts`.

## Testing Guidelines
- Vitest is the runner. Place tests in `tests/` as `*.test.ts`. Keep unit tests offline-friendly (no live Hedera/mirror calls).
- Cover config precedence, CSV escaping, HMAC verification, cap math, and tool/plugin shape. On-chain flows are exercised manually against testnet, not in the unit suite.

## Architecture Notes
- The HCS topics are the source of truth — there is no database. `store.json` only caches topic IDs and a fast-path idempotency guard (see below).
- `pay_on_merge` executes the transfer inside `coreAction` (not a two-phase `secondaryAction`) because it is a read-policy → resolve-identity → transfer → seal flow, not a simple build-then-submit. Idempotency uses two layers: an instant local guard in `store.json` plus the durable RECEIPTS topic.

## Commit & Pull Request Guidelines
- Use Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`) with scopes where useful (`feat(tools): …`).
- PRs should include a summary, test commands/results, and any config/env changes. Update `README.md` and `docs/` when tool behavior or environment variables change.

## Security & Configuration Tips
- Never commit secrets. Use environment variables (`HEDERA_*`, `GEMINI_API_KEY`, `GITHUB_WEBHOOK_SECRET`, `SLACK_WEBHOOK_URL`); document new keys in `.env.example` and `docs/CONFIGURATION.md`.
- The webhook validates `X-Hub-Signature-256` before processing any payload. The PR merge is the human approval; the POLICIES cap is the financial control.
